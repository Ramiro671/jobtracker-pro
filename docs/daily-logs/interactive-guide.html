✅

🔐

API Layer en detalle

JobApplicationsController.cs — CRUD de aplicaciones. Cada endpoint extrae el userId del JWT y crea un Command/Query para MediatR.

AuthController.cs — Register, Login, Refresh, Logout. Rate limited a 10 req/min.

ExceptionHandlingMiddleware.cs — Captura excepciones no manejadas y las convierte a ProblemDetails (RFC 9110). ValidationException → 400, UnauthorizedAccessException → 401.

// Controller delega a MediatR [HttpPost] public async Task<IActionResult> Create( [FromBody] CreateJobApplicationRequest req) { var result = await _sender.Send( new CreateJobApplicationCommand(...) ); return CreatedAtAction(..., result); }

Application Layer en detalle

Commands = operaciones que cambian estado (Create, Update, Delete, ChangePassword). Implementan IRequest<T>.

Queries = operaciones de lectura (GetJobApplications). No mutan estado.

Handlers = logica que ejecuta el Command/Query. Accede a repos via interfaces.

Validators = reglas FluentValidation. Se ejecutan ANTES del handler via pipeline behavior.

DTOs = objetos de transferencia de datos. JobApplicationDto tiene Status como int (no string) para que el frontend pueda comparar numeros.

// Command + Handler CQRS public record CreateJobApplicationCommand( Guid UserId, string Title, string CompanyName, string? JobUrl ) : IRequest<JobApplicationDto>; public class CreateJobApplicationHandler : IRequestHandler<CreateJobApplicationCommand, JobApplicationDto> { public async Task<JobApplicationDto> Handle(CreateJobApplicationCommand req, ...) { /* logica aqui */ } }

Infrastructure Layer en detalle

ApplicationDbContext — DbContext de EF Core. Define DbSets (tablas). Configura relaciones, índices, conversiones.

Repositories — Implementan IJobApplicationRepository, IUserRepository, etc. Solo la infraestructura sabe de EF Core.

JwtTokenGenerator — Genera y firma JWT con clave secreta. El token incluye userId como claim "sub".

RedisCacheService — Implementa ICacheService. Try/catch en cada operacion para no romper el flujo si Redis esta caido.

// EF Core DbContext public class ApplicationDbContext : DbContext { public DbSet<User> Users => ...; public DbSet<JobApplication> JobApplications => ...; public DbSet<Company> Companies => ...; }

Domain Layer en detalle — EL CORAZON

Entidades = objetos con identidad (Id unico): User, JobApplication, Company, RefreshToken.

Metodos de dominio = la logica vive en la entidad, no en el controlador:

— JobApplication.UpdateStatus() — cambia status y registra appliedAt automaticamente.

— User.ChangePassword() — reemplaza el hash de contrasena.

Sin dependencias externas = no referencia EF Core, ASP.NET, ni nada externo. Solo .NET base class library.

// Entidad con logica de dominio public class JobApplication { public Guid Id { get private set; } public ApplicationStatus Status { get private set; } public void UpdateStatus( ApplicationStatus newStatus, string? notes = null) { if (newStatus == ApplicationStatus.Applied && AppliedAt == null) AppliedAt = DateTime.UtcNow; Status = newStatus; UpdatedAt = DateTime.UtcNow; } }
Regla de dependencia: Domain no conoce a nadie. Application conoce Domain. Infrastructure conoce Domain + Application (implementa sus interfaces). API conoce a todos. Si inviertes esta regla, el proyecto se vuelve imposible de testear.

Autenticacion con JWT

JSON Web Token es un estandar para transmitir informacion de forma segura. El backend firma el token, el frontend lo guarda y lo envia en cada request.

Estructura del JWT

Un JWT tiene 3 partes separadas por puntos. Haz clic en cada parte para ver su contenido:

Header
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
Payload
eyJzdWIiOiJ1c2VyLWd1aWQiLCJlbWFpbCI6Ii4uLiJ9
.
Signature
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Header (Encabezado)

Indica el algoritmo de firma y tipo de token. Codificado en Base64URL (no encriptado — solo codificado).

{ "alg": "HS256", "typ": "JWT" }

Payload (Carga util)

Contiene los "claims" — datos del usuario. El backend extrae el userId del claim "sub" en cada request autenticado. NO guardes datos sensibles aqui — es solo Base64, no encriptado.

{ "sub": "3fa85f64-5717-4562-b3fc-2c963f66afa6", // userId "email": "user@example.com", "name": "John Doe", "iss": "JobTrackerPro", // issuer "aud": "JobTrackerPro", // audience "exp": 1735689600, // expira en 60 min "iat": 1735686000 // emitido en }

Signature (Firma)

HMAC-SHA256 del header + payload, firmado con la clave secreta del servidor. Si alguien modifica el payload, la firma no coincide y el token es rechazado.

HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload), secretKey // JwtSettings:Secret en appsettings.json )

Flujo de Login + Refresh Token

1. Login

POST /api/auth/login → el backend valida credenciales, genera:

— Access Token (JWT, expira en 60 min)

— Refresh Token (GUID, expira en 7 dias, guardado en DB)

El frontend guarda ambos en localStorage.

2. Refresh automatico

Cuando el access token expira, el API devuelve 401. El interceptor de Axios:

1. Detecta el 401 y llama POST /api/auth/refresh

2. Encola todos los requests pendientes

3. Cuando llega el nuevo token, reintenta todos

4. Si el refresh tambien falla → logout

// client.ts — Interceptor con queue let isRefreshing = false; let pendingQueue: Array<{resolve, reject}> = []; apiClient.interceptors.response.use(null, async (error) => { if (error.response?.status === 401 && !config._retry) { if (isRefreshing) { return new Promise((resolve, reject) => pendingQueue.push({ resolve, reject }) ); } isRefreshing = true; config._retry = true; const { accessToken } = await refresh(); drainQueue(null, accessToken); // reintenta todos isRefreshing = false; return apiClient(config); // reintenta el original } });

Application Layer — CQRS y MediatR

CQRS (Command Query Responsibility Segregation) separa las operaciones de escritura (Commands) de las de lectura (Queries). MediatR es el "bus" que conecta quien pide con quien ejecuta.

Command (escribe)

Cambia el estado del sistema. Puede devolver un resultado o nada (Unit).

// Sin retorno public record ChangePasswordCommand( Guid UserId, string CurrentPassword, string NewPassword ) : IRequest; // IRequest sin <T> // Con retorno public record CreateJobApplicationCommand(...) : IRequest<JobApplicationDto>;

Query (lee)

Solo lee datos, no modifica estado. Siempre devuelve algo.

public record GetJobApplicationsQuery( Guid UserId ) : IRequest<List<JobApplicationDto>>; // Handler public class GetJobApplicationsHandler : IRequestHandler<GetJobApplicationsQuery, List<JobApplicationDto>> { public async Task<...> Handle(...) { ... } }

Pipeline de MediatR

Antes de llegar al Handler, el request pasa por "behaviors" (middleware de MediatR):

HTTP Request ↓ Controller → _sender.Send(command) ↓ ┌─────────────────────────────────┐ │ MediatR Pipeline │ │ │ │ 1. ValidationBehavior │ ← FluentValidation ejecuta aqui │ └─ Si invalido: throw 400 │ │ │ │ 2. Tu Handler │ ← Logica de negocio │ └─ Accede a repos/DB │ │ │ └─────────────────────────────────┘ ↓ Controller → return resultado

Validators (FluentValidation)

public class CreateJobApplicationValidator : AbstractValidator<CreateJobApplicationCommand> { public CreateJobApplicationValidator() { RuleFor(x => x.Title) .NotEmpty().MaximumLength(200); RuleFor(x => x.CompanyName) .NotEmpty().MaximumLength(200); RuleFor(x => x.UserId) .NotEmpty(); } }

DTOs — Por que no devolver la entidad directamente?

Problema con entidades directas

— Expones campos internos (PasswordHash)

— Cambiar la entidad rompe la API publica

— Ciclos de referencia en JSON

— Los enums se serializan como string por defecto (quieres int)

Solucion con DTOs

public class JobApplicationDto { public Guid Id { get set; } public string Title { get set; } public string CompanyName { get set; } // Status como int, NO como string! public int Status { get set; } public DateTime? UpdatedAt { get set; } }

Infrastructure Layer

Implementaciones concretas de las interfaces definidas en Application/Domain. Aqui vive todo lo que toca el mundo exterior: DB, cache, email, tokens.

EF Core — Repository Pattern

Patron Repository: oculta la implementacion de acceso a datos. El handler solo conoce la interfaz IJobApplicationRepository, no sabe si es PostgreSQL o MongoDB.

// Interfaz en Domain public interface IJobApplicationRepository { Task<List<JobApplication>> GetByUserIdAsync(Guid userId, ...); Task AddAsync(JobApplication app, ...); Task UpdateAsync(JobApplication app, ...); Task DeleteAsync(Guid id, ...); } // Implementacion en Infrastructure public class JobApplicationRepository : IJobApplicationRepository { private readonly ApplicationDbContext _db; public async Task<List<JobApplication>> GetByUserIdAsync(Guid userId, ...) => await _db.JobApplications .Where(a => a.UserId == userId) .ToListAsync(ct); }

Unit of Work: agrupa multiples operaciones en una sola transaccion. Si algo falla, todo se revierte.

public interface IUnitOfWork { Task<int> SaveChangesAsync(...); } // En el handler: await _repository.AddAsync(app, ct); await _unitOfWork.SaveChangesAsync(ct); // EF Core hace 1 solo INSERT en DB

Migrations: versionado del schema de la DB. En lugar de alterar la DB manualmente, EF genera archivos de migracion que se aplican automaticamente al iniciar la API.

dotnet ef migrations add NombreMigracion dotnet ef database update // O automaticamente en Program.cs: db.Database.Migrate();

Redis Cache — Con fallback graceful

public class RedisCacheService : ICacheService { public async Task<T?> GetAsync<T>(string key, ...) { try { var json = await _cache.GetStringAsync(key, ct); return json == null ? default : JsonSerializer.Deserialize<T>(json); } catch (Exception ex) { // Si Redis no esta disponible, solo loggea // y continua sin cache (no rompe el flujo) _logger.LogWarning(ex, "Redis unavailable"); return default; } } }

StaleNotificationService — Background Job

// Se ejecuta cada 24h en background public class StaleNotificationService : BackgroundService { protected override async Task ExecuteAsync(CancellationToken ct) { while (!ct.IsCancellationRequested) { await CheckAndNotifyAsync(ct); await Task.Delay(TimeSpan.FromHours(24), ct); } } // Busca apps con 7+ dias sin actividad // Agrupa por usuario → envia 1 email por usuario // Usa IServiceScopeFactory porque DbContext es scoped // y BackgroundService es singleton }
